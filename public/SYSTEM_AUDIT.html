Admin actions pattern

All admin operations follow this pattern:

  1. Client-side action (in app/actions/*.ts)
  2. Direct Supabase call (using anon key + RLS)
  3. RLS enforces authorization (via is_admin() function)
  4. Audit log written after successful operation

Example flow (admin/src/app/actions/user-actions.ts)

export async function toggleUserSuspension(userId, currentStatus, reason) {
  // Direct Supabase update - RLS checks is_admin()
  await supabase.from('profiles').update({
    is_suspended: !currentStatus
  }).eq('id', userId);

  // Log the action
  await logAdminAction('suspend_user', 'user', userId, { reason });
}

RLS (row level security) policies

Security model

  • All tables have RLS enabled
  • Policies check auth.uid() for ownership
  • Admin bypass via is_admin() function
  • is_admin() checks profiles.is_admin = true

Key RLS patterns

Profiles
SELECT: auth.uid() = id OR is_admin()
UPDATE: auth.uid() = id OR is_admin()
-- protect_profile_metadata trigger prevents non-admins from changing governance fields

Trips
SELECT: public OR is_admin()
INSERT: auth.uid() = traveler_id
UPDATE: auth.uid() = traveler_id OR is_admin()
DELETE: auth.uid() = traveler_id OR is_admin()

Shipments
SELECT: public OR is_admin()
INSERT: auth.uid() = sender_id
UPDATE: auth.uid() = sender_id OR is_admin()
DELETE: auth.uid() = sender_id OR is_admin()

Offers
SELECT: driver_id = auth.uid() OR shipment.sender_id = auth.uid() OR is_admin()
INSERT: driver_id = auth.uid()
UPDATE: (via RPCs only)
DELETE: is_admin()

Bookings
SELECT: requester_id = auth.uid() OR traveler_id = auth.uid() OR is_admin()
INSERT: requester_id = auth.uid()
UPDATE: requester_id = auth.uid() OR traveler_id = auth.uid() OR is_admin()

FSM (finite state machine) triggers

  • enforce_booking_state_machine() — Prevents illegal booking status transitions
  • enforce_trip_state_machine() — Prevents illegal trip status transitions (if exists)
  • enforce_shipment_state_machine() — Prevents illegal shipment status transitions (if exists)

  • sync_trip_load() — Auto-updates trips.current_load_kg when bookings change
  • Unique index uq_offers_one_accepted_per_shipment — Only one accepted offer per shipment

Audit triggers

Metadata protection

  • protect_profile_metadata() — Prevents non-admins from changing governance fields (is_admin, is_suspended, strike_count, etc.)

Core business flows

1. Shipment + offer flow (NEW — post migration 00021)

Sender journey

  1. Sender posts shipment → shipments table (status: pending_approval or pending)
  2. Admin approves (if needed) → status: pending
  3. Drivers see shipment in search
  4. Drivers send offers → send_offer() RPC → offers table (status: sent)
  5. Sender views offers in shipment details
  6. Sender accepts one offer → accept_offer() RPC:
    • Offer status: sent → accepted
    • Other offers: sent → rejected (reason: other_offer_accepted)
    • Shipment status: pending → accepted
  7. Handover: Sender marks goods handed over; driver confirms goods received → shipment in_transit
  8. Delivery: With OTP: driver enters delivery code → completed. Without OTP: driver marks delivered → sender confirms → completed
  9. Payment (optional): Sender marks payment sent; driver confirms payment received
  10. Rating exchange

Driver journey

  • Search for shipments → send offer with price + message via send_offer_with_message()
  • Wait for sender response; if accepted → pickup/delivery lifecycle, payment, rate sender

2. Trip + booking flow (LEGACY — being phased out)

  • Driver posts trip → trips (pending_approval or available)
  • Admin approves → available; senders book → bookings (pending)
  • Driver accepts → accept_booking_offer() RPC: booking accepted, siblings other_accepted, trip booked, load updated via trigger
  • Similar handover/delivery/payment flow as shipments
Note: Migration 00021 removed bookings.shipment_id — bookings are trip-only. The offer system replaced shipment bookings.

3. Chat / messaging flow

  • Booking chat (LEGACY): messages.booking_id — requester ↔ traveler
  • Offer chat (NEW): messages.offer_id — sender ↔ driver; first message via send_offer_with_message()
  • Support: messages.ticket_id — user ↔ admin

4. Verification flow

Driver

  • Register as driver → traveler_status: 'pending'; upload documents → admin reviews in /drivers
  • Approve/reject via updateVerificationStatus(); if approved: traveler_status: 'approved', is_driver: true

Company

  • Register → company_status: 'pending', account_type: 'company'; documents; admin on /companies
  • If approved: company_status: 'approved'

5. Moderation flow

  • Flagging: Admin flags shipment/trip → is_flagged, moderation_status: 'pending_review'
  • Resolve: Cleared / removed / escalated with corresponding status updates
  • Governance: Suspension, blocking, freezing, strikes, soft delete (deleted_at)

6. Notification flow

  1. App registers FCM token → register_notification_token()
  2. Event occurs → sendNotificationToUser(): writes notifications, calls Edge Function send-push-notification
  3. FCM delivery → app navigates to relevant screen

Risk & fragility areas

1. Sensitive lifecycle transitions

  • Shipment: pending→accepted (atomic sibling rejections); accepted→in_transit; in_transit→delivered/completed — payment & finality
  • Trip: available→booked; booked→in_transit→completed
  • Booking: race on accept; other_accepted siblings; FSM forbids certain reversions from in_transit/delivered/completed

2. Critical RPC functions

High risk: accept_offer(), accept_booking_offer() — concurrency; mitigations: FOR UPDATE, unique indexes, status checks. send_offer_with_message() — must stay transactional. admin_provision_profile() — orphan risk; rollback in Edge Function.
Medium risk: send_offer() spam if not rate-limited; cancel_offer/reject_offer guarded by status checks.

3. Dangerous admin operations

  • Destructive: deleteShipment(), deleteTrip(), setUserBlocked(), updateUserGovernance() — safeguards: status checks, reasons, audit, RLS/triggers
  • State overrides: admin cancel/reopen/approve/reject shipment & trip actions

4. RLS security concerns

  • Admin granularity unified to is_admin() (migration 00053); planned tightening (e.g. 00049)
  • Offers: no direct user UPDATE (RPC-only — good)
  • SECURITY DEFINER RPCs must validate auth.uid()

5. Frontend assumption risks

  • Flutter: offline sync staleness; optimistic UI; deep-link dedup window
  • Admin: no realtime (stale UI); client-side actions depend on RLS; no optimistic updates (slower but predictable)

6. Performance-sensitive screens

  • Flutter: home (many sources); search RPCs; chat realtime (throttle ~300ms, distinct); batched offer statuses
  • Admin: user list pagination; analytics dashboard_stats(); large audit log — pagination/filters

7. Stale state risks

  • Admin: no realtime — manual refresh
  • Flutter: subscriptions scoped to active screens; SyncWorker / cache TTL considerations
